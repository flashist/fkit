## 1. Executive summary

**Headline verdict: the "usable in ANY software project" claim is overstated.** What fkit actually is — and is genuinely good at — is narrower: a zero-dependency, well-tested **two-CLI (Claude Code + Codex) skill-distribution and workflow-conventions kit for a solo developer in a git repo with GitHub network access**, carrying one piece of genuinely novel process IP (the review-ledger loop-resistance discipline). Each step away from that configuration — a team, a single CLI, a third provider, a monorepo, a non-git VCS, an air-gapped network — hits a concrete, citable break rather than graceful degradation. The honest claim is "true with heavy caveats," and the caveats are currently documented nowhere.

**Top findings, most important first:**

1. **The multi-agent story rests on an unverified alpha port the README never mentions.** The agent team with a consultation topology exists only under `omnigent/`; every tool edge is self-flagged "(Alpha: verify …)", the only structural guardrail (`omnigent.inner.nessie.policies.blast_radius`) is an external symbol defined nowhere with *contradictory* documented semantics across configs, all six agents run `sandbox: none`, and no test or run artifact validates any of it. Every role boundary — including the reviewer's "never edit source" — is prose over a fully-permissioned substrate. (`omnigent/*/config.yaml`; README.md)

2. **No work-carrying edges exist; the human is the pipeline.** There is no producer→coder, coder→reviewer, or reviewer→coder edge. The declared tools carry consultations and wiki lookups; actual work moves through files with the owner launching each agent at every stage. Combined with the single named `{{owner}}` gating every commit, task move, plan approval, and review disposition, fkit is structurally single-seat. "Agents delegate/consult each other through a defined topology" (brief §2) overstates what the wiring does.

3. **The operating layer is missing — install is minutes, operation must be reverse-engineered.** Mechanical setup is genuinely excellent (zero deps, sub-second, offline-capable core). But no task-brief or sprint-plan template exists anywhere, no worked example of a brief/sprint/ledger/wiki page ships, no document connects plan → implement → review → done, and the producer is ordered to "follow the established format in `ai-agents/tasks/` exactly" against directories that ship only `.gitkeep`. This is now the single biggest adoption blocker. (`generic/roles/producer.preset.md:47,63`; `generic/ai-agents/tasks/`)

4. **Setup and supply chain have real safety defects.** The install happy path runs `bootstrap --force`, which unconditionally overwrites a legacy project's hand-written `CLAUDE.md`, `AGENTS.md`, `PROJECT.md`, wiki skeleton, and tuned roles (probe-confirmed data loss). The manifest is parsed by a hand-rolled YAML subset with **zero validation** — ordinary YAML yields exit 0 and corrupted artifacts (probe: project named `[unclosed` propagated into CLAUDE.md). And every consumption path runs `npx --yes github:flashist/fkit` **unpinned against default-branch HEAD**: at review time users are executing 6 unreleased commits self-identifying as a version that was never tagged. Delegation stubs default to headless `claude -p --permission-mode acceptEdits`, silently erasing the review-only/approval contracts of the skills they wrap. (`bin/bootstrap.mjs:34-37,68,97-108`; `bin/lib.mjs:42-68`; `skills/fkit/SKILL.md:9,90`; `bin/compile-skills.mjs:137-140`)

5. **What genuinely works must be preserved.** The review-ledger loop-resistance model (accepted residuals with "re-raise only if", defect-vs-frontier classification, output-side dedup) is the kit's crown jewel and is model-agnostic. The exactly-one-owner routing invariant, the `fkit:generated` marker discipline, idempotent config migrations, the regenerated `config-schema.json`, and the zero-dependency hermetic test style are all verified strengths. So is the reviewer↔coder shared-ledger contract — the one place where multi-agent coordination is specified precisely enough to survive stateless sessions.

#### 1b. Portability scenarios

- **Solo vs. team — [HIGH · tradeoff]:** structurally single-seat. One singular `owner` in the manifest; `task-done` "runs only when {{owner}} invokes it"; every approval routes through one person; and because nothing commits, all coordination state (task moves, sprint edits, ledgers) is invisible to teammates while wiki-sync reads only committed history. No multi-owner or delegation-of-approval concept exists anywhere, and the single-owner scope is not documented as deliberate. (`manifest/ai-agents.schema.yml:18`; `generic/skills/task-done/skill.md:19-20,50-51,99`)
- **Non-JS stack — fine, with a caveat:** skill content is language-agnostic; the Node ≥ 18 toolchain plus GitHub npx access is a hard prerequisite a Rust/Go/Python shop must accept. Minor fingerprint: `process-review` hard-codes "Cite `src/...`" (`generic/skills/process-review/skill.md:124`).
- **Monorepo — [MEDIUM · gap]:** no story at all. One project identity per manifest, `CLAUDE.md`/`AGENTS.md`/`.codex/config.toml` at the `--out` root, root-relative `ai-agents/` paths in skills, stubs that must run "from the project root". Nothing crashes; nothing is addressed. (`manifest/ai-agents.schema.yml:15-21`; `bin/bootstrap.mjs:97-118`; `bin/compile-skills.mjs:160`)
- **Non-git VCS — breaks** (see table above). On hg/Perforce the written procedures fail; jj works only colocated.
- **Single-model / other provider — [HIGH · defect]:** Claude-only users get a broken default (seeded Codex pins → dead-end stubs). Codex-only is worse: `stateful-review`'s body hardcodes Claude Code's `code-reviewer` agent, `/code-review high`, `AskUserQuestion`, and a `~/.claude/plugins/cache/openai-codex/...` companion-script glob complete with workarounds for that plugin's bugs — it cannot run natively on Codex. A third provider (Gemini, Cursor, Copilot) is impossible without editing kit source. (`generic/skills/stateful-review/skill.md:34-59,120`; `bin/lib.mjs:102`)
- **Greenfield vs. legacy:** greenfield hits the cold-start gap (producer init reads a sprint plan and queries a wiki that no mechanism ever populated — mitigated by the query skill's explicit gap-flagging, so degraded rather than broken). Legacy hits a concrete hazard: bootstrap step 4 writes `CLAUDE.md`/`AGENTS.md` with **no exists-check**, silently replacing a legacy project's hand-written CLAUDE.md — and the entry skill's own flow passes `--force`. **[HIGH · defect]** (`bin/bootstrap.mjs:97-100`; `skills/fkit/SKILL.md:90`)

#### 1c. Adoption cost

- **Prerequisites (real ones):** Node ≥ 18, network to one personal GitHub repo, npx, and — for routing to deliver value — both paid agent CLIs authenticated. The machinery itself is featherweight (zero deps, no git/network when cloned).
- **[HIGH · strength] Mechanical setup is genuinely cheap.** Probe-confirmed: bootstrap of a non-git directory in <1 s; full round-trip `config set → sync → stub change` worked exactly as documented; 66/66 tests pass bare. Time-to-first-scaffold is minutes, dominated by the interview.
- **[MEDIUM · defect] The README's human setup path is broken as written.** Pass 2 of the documented two-pass bootstrap always exits 1 (pass 1 created `ai-agents/`; bootstrap refuses without `--force`, which neither README nor bootstrap's own printed next-step mentions). Also: README claims `config show` is read-only — it actually creates/migrates `config.json` and always rewrites `config-schema.json` (the kit's own test asserts this); `sync --kit` is advertised but parsed nowhere and silently ignored; `kit_version` in the manifest is documented as "sync checks it" but read by no script. (`README.md:46-48,61-62`; `bin/bootstrap.mjs:34-37,57`; `bin/sync.mjs:9,17-26`; `tests/config-cli.test.mjs:53-71`)
- **[HIGH · defect] Zero manifest validation.** The hand-rolled YAML-subset parser silently drops any line without a colon and keeps trailing `#` comments inside values. Probe: exit 0 with project name `[unclosed` propagated into CLAUDE.md, `model = ""` in `.codex/config.toml`, and an inline comment baked into compiled skill descriptions. The subset is documented only in the entry skill and schema file — not in the starter manifest users actually edit. (`bin/lib.mjs:42-68`)
- **[MEDIUM · gap] Concept load ~14, of which ~6 earn first value.** Several onboarding concepts are dead on arrival: the manifest `skills:` block is consumed at pass 1 before the user can edit it; `kit_version`, `entry_skill`, and `knowledge` are read by no script; declared roles beyond producer scaffold nothing, silently. The starter manifest is the fictional "Nimbus" sample verbatim, fake model ids and all. (`bin/bootstrap.mjs:49-55,81-87`; `examples/sample.ai-agents.yml:1-9`)

**Task 1 verdict: over-stated.** As scoped today, the claim that holds is: *"usable in any git-based, network-connected, single-project repo operated by one person using Claude Code and/or Codex, for the pre-merge slice of the lifecycle."* Each qualifier in that sentence is load-bearing, and none of them appears in the README.

All six agents were reviewed individually against their configs, skills, the other five configs, and their generic-kit counterparts.

#### fkit-producer — earns its place; wrong-shaped skill set

The apparent contradiction between "must not move task files" and its two file-moving skills is explicitly reconciled in-prompt (owner-invocation-only exception) — handled, not a defect. **[strength]** (`omnigent/fkit-producer/config.yaml:102-106,184-189`; `omnigent/fkit-architect/config.yaml:187-191`)

The prompt orders "follow the established format in `ai-agents/tasks/` exactly" against directories that ship `.gitkeep` only; no brief/sprint template exists anywhere in the kit. On a fresh project the producer cannot follow its own instructions. (`config.yaml:162-166,205-207`; `generic/roles/producer.preset.md:47,63`)
- **[MEDIUM · defect] Mandatory init depends on artifacts nothing creates** (governance wiki pages, `plan-sprint-N.md`) — degrades gracefully via the query skill's gap-flagging, but wastes delegation round-trips and lacks a cold-start branch ("no sprint plan exists — offer to create one"). #### fkit-coder — the strongest agent
- **[HIGH · strength] Sole source-write authority with the port's best-drawn boundaries.** Every other agent prohibits source edits (architect's narrow stubs-only carve-out is explicit and disciplined). - **[HIGH · defect] Fuzzy task-id can silently fork the shared ledger.** Both coder and reviewer independently "infer" the task-id (branch / diff / task file); no canonical derivation rule exists anywhere; both sides auto-create the ledger on a miss. Divergent ids split `ai-agents/reviews/<task-id>.md` and silently defeat the accepted-residuals loop prevention — the design's whole point. #### fkit-reviewer — best-designed persona; forked dialect
- **[strength] Coherent REVIEW-ONLY discipline** ("never edit source — not even with approval"), graceful degradation done right ("never pair a clean 'Ready to merge' with a missing reviewer"), clean skill split (ephemeral `review` vs. ledger-backed `stateful-review`).
- **[HIGH · defect] Incompatible ledger schema at the same path.** The omnigent stateful-review writes *Reviewer findings / Coder response / Accepted residuals* to `ai-agents/reviews/<task-id>.md`; the generic kit's ledger at the **same path** is *Accepted residuals / Decision log / Open-actionable*. Hand-maintained, no shared source; a project running both the compiled kit skills and the omnigent agents collides two schemas on one file — corrupting the kit's most valuable mechanism, and contradicting the single-sourcing premise. (Softener: the residuals entry structure is identical in both dialects.) (`omnigent/fkit-reviewer/skills/stateful-review/SKILL.md:29-51`; `generic/ai-agents/reviews/README.md:50-70`)
- **[MEDIUM · tradeoff] Triple verification, verdicts never persisted.** Findings can be verified three times per round (sidekick, reviewer Step 3, coder Steps 3–3.5); the shared-doc schema persists severity but not the reviewer's verdict or defect/frontier call, and coder independence ("never inherit the reviewer's label") is deliberate — but no rule reconciles a reviewer-report vs. coder-ledger disagreement beyond implicit owner arbitration.
- **[LOW · defect]** The ephemeral `review` skill never operationalizes the residual-dedup its own prompt promises, and offers four verdicts where the prompt lists five. **[MEDIUM · gap]** No architect consult edge, despite Step 3 explicitly asking whether findings "are missing project context (architecture …)" — narrow gap (it can read ADR files directly; only unrecorded design intent is unreachable).

#### fkit-adversarial-reviewer — cleanest contract; the invariant-breaker
- **[HIGH · strength] The sub-agent promotion is a genuine improvement.** It replaces the generic skill's brittle plugin plumbing (companion-script glob, zombie-job cancellation lore, 2-minute-timeout warnings) with one declared tool call; the findings-only output contract exactly matches what the lead consumes; anti-fabrication rules ("an honest 'clean' is a valid result") are excellent. The brief's "everyone → fkit-wiki" is false for one of five consumers, and four configs' "all four non-wiki agents" comment papers over it. (`omnigent/fkit-wiki/config.yaml:79-83`)

Identical YAML, contradictory documentation of the fleet's *only* structural safety mechanism. - **[MEDIUM · defect] Content defects inside the skills:** the `architecture` ingest keyword circularly targets `CLAUDE.md` + the wiki's own `schema.md` instead of the `knowledge-base/architecture.md` the architect's inspect actually produces (the handoff still works via three other routes — the keyword is just wrong); sync hardcodes `wiki/decisions/cancelled-tasks.md`, a page nothing in the repo defines — an origin-project leak. (`skills/ingest/SKILL.md:20`; `skills/sync/SKILL.md:44`)
- **[MEDIUM · gap] All four SKILL.md files are hand-flattened, marker-less copies** of `generic/skills/fkit-wiki-*` (diff-verified: near-verbatim today), hardcoding the wiki path the generic kit keeps configurable — a manual drift channel inside a kit whose premise is single-sourced skills. The port *is* documented at agent level (config header), but no per-file marker or sync check exists.

#### The consultation topology as a whole
- **Verified graph:** producer→{wiki, architect}, coder→{wiki, architect}, reviewer→{wiki, adversarial}, architect→{wiki, producer}; wiki and adversarial are tool-less leaves. Matches brief §2 **except** "everyone → fkit-wiki" (false for the adversarial reviewer).
- **[HIGH · defect] Suggested, not enforced.** All six agents: full filesystem/shell, `sandbox: none`. The reviewer's header concedes REVIEW-ONLY is "enforced in the prompt + skills, not the sandbox." Any agent can mechanically read the wiki, edit code, or shell out to a peer. The one structural guardrail is an opaque external symbol with contradictory documented semantics; every edge depends on `../`-relative config resolution that every tools block flags "(Alpha: verify)". If that resolution fails, the entire graph is dead — and only reviewer→adversarial has a scripted degradation.
- **[MEDIUM · defect] The producer⇄architect cycle is structurally unbounded.** `max_sessions: 1` caps sessions per instance, not recursion depth — each spawned sub-agent gets its own tools, so producer→architect→producer′→architect′… is structurally possible (also reachable via coder→architect→producer). Prevention is a YAML comment ("Loop-safe: focused consult, no counter-consult") plus prompt prose. There is also no token/turn budget anywhere, so a consult chain has no cost ceiling.
- **[MEDIUM · tradeoff] The wiki gateway: single-writer strength, expensive and failure-blind read chokepoint.** One schema-enforcing writer is a defensible design. But every delegated lookup boots a fresh session that re-reads schema/index/log; producer and coder must treat answers as **ground truth**, so stale or hallucinated wiki content propagates *with authority*; and no caller has a wiki-unavailable degradation clause — in pointed contrast to the reviewer's explicit sidekick handling. Mitigations exist gateway-side (query cites sources, flags gaps) but consumers never verify citations.
- **[HIGH · tradeoff] No edge carries work** (Finding 2 in the executive summary). The reviewer↔coder file-mediated contract is excellent precisely because it *doesn't* need an edge — but end-to-end flow is the human launching agents in sequence, and this is documented nowhere.

### Task 3 — Critique

#### Failure modes (grounded in file text)

- **[HIGH · defect] Headless delegation stubs erase per-skill safety contracts.** `stateful-review` forbids all source edits ("not even with your approval") and gates ledger writes on interactive `AskUserQuestion`; its compiled Codex-side stub delegates via `claude -p --permission-mode acceptEdits` — non-interactive, auto-edit, nobody present to answer the mandatory questions. The uniform `DEFAULT_EXEC` transport cannot express read-only, interactive, or non-delegable skills. (`generic/skills/stateful-review/skill.md:10,120`; `.codex/skills/stateful-review/SKILL.md:16`; `bin/compile-skills.mjs:137-140`)
- **[MEDIUM · defect] Stub commands are shell-mangled and injection-prone.** The Claude-side stubs for Codex-owned skills embed `$fkit-wiki-*` inside a double-quoted string (bash expands `$fkit` to empty — live-verified), and raw user arguments are spliced into the same double-quoted command with no quoting guidance, so arguments containing `$`, quotes, or backticks can break the command or execute locally first. (`.claude/skills/fkit-wiki-ingest/SKILL.md:17,20`; `bin/compile-skills.mjs:160-167`)
- **[HIGH · tradeoff] The never-commit discipline collides with its own machinery at team scale.** The rule is stated per-skill and per-config but absent from the generated `CLAUDE.md`/`AGENTS.md` — an agent doing ordinary coding never sees it. One owner serializes everything; and since wiki-sync reads only committed history, never-commit makes the owner the knowledge-freshness bottleneck: sync truthfully reports "Wiki is up to date" over a tree of uncommitted done-task briefs (freshness is delayed, not lost — the watermark doesn't advance on an empty delta, and force-mode/ingest bypass git).
- **[HIGH · gap] Concurrency is unaddressed.** Reviewer and coder deliberately co-edit the same ledger under prose-only section ownership; task skills grep-edit shared sprint tables; ingest/sync append to index/log — all uncommitted in one working tree, with fuzzy task-id derivation able to fork a ledger. No lock, no worktree convention, no merge guidance. (`omnigent/fkit-reviewer/skills/stateful-review/SKILL.md:53-60`)
- **[MEDIUM · tradeoff] Config drift by design.** `config set` doesn't recompile ("no visible effect until sync runs"); the restart signal compares kit versions only, so a same-version ownership flip reports `restart-required: no` while live sessions execute the stale variant. (`generic/skills/fkit-config/skill.md:81`; `bin/compile-skills.mjs:288-298`)
- **[MEDIUM · gap] Wiki answers are "ground truth" with no failure fallback** — the only handled branch is "found nothing"; a gateway error or wrong answer has no protocol, and consumers never verify the citations the gateway provides. Contrast the review path, which mandates verification of every reviewer claim.
- **[MEDIUM · gap] Token/cost blow-ups have no mechanism anywhere.** Real local mitigations exist and deserve credit (pass_history: false everywhere, 2-hop query cap, delta-based sync, "Do not delegate further" in stubs, ledger dedup). But the default topology multiplies spend by design — every agent's init mandates wiki round-trips through a cold-booting sub-agent; both review skills run two models; stubs spawn full cold-start second-model sessions — and there is no budget, cap, or spend-observability concept in the entire kit (repo-wide grep: the only spend awareness is a warning that probing the Codex companion "launches a real, billed run").

#### Conceptual integrity & documentation

- **[MEDIUM · tradeoff] ~14 concepts, accreting legacy at v0.1.** The manifest schema spends nearly half its 70 lines explaining what no longer exists (`routing:` removed, `skills:` dead seed, abolished `both`); the starter manifest ships a block that is already dead when the user first sees it. The core trio — ledger, routing, wiki — is individually coherent and learnable; the legacy vocabulary is not.
- **[MEDIUM · tradeoff] The stub system is the heaviest machinery for the smallest problem** — and cannot express its own skills' claims: `fkit-config`'s source says it "never delegates" while its compiled Codex output is a delegation stub saying "Do not do the work yourself." Meanwhile the genuinely load-bearing layer — operating documentation — is missing (Finding 3).
- **[MEDIUM · defect] The release tooling fails its one job.** A 221-line release script syncs only `VERSION` and `package.json`; the repo shows four version surfaces disagreeing: README "v0.1.0", `plugin.json` 0.1.0, compiled skill markers 0.1.17, `VERSION` 0.1.18. `release.mjs` also stages *everything* (`git add -A`) — a documented foot-gun. (`bin/release.mjs:144-151,183`)
- **[LOW · defect] Terminology drifts:** roles are "role-agents"/"terminal-tab personas"/"the agent roster" across surfaces; `both` is simultaneously invalid (skills), documented-valid (roles schema), used (sample), and silently coerced to `claude` (`bin/scaffold-role.mjs:41`).
- **Discoverability verdict:** a newcomer could *install* fkit from the README but could not *operate* it — no document connects brief → plan → implement → review → done, and the omnigent port (per the brief, the fullest expression) is mentioned nowhere in the README.

### Task 4 — What's strong (preserve these)

Every item below was re-verified in the files; two commonly-praised candidates were **rejected** (noted at the end).

1. **The review-ledger loop-resistance model — the crown jewel. [HIGH · strength]** Accepted residuals with explicit "Re-raise only if" conditions; mandatory defect-vs-frontier-move classification *before* verdicts; verifier-owned severity; output-side dedup as the guarantee (priming is best-effort); Partial-verdict precedence; proactive convergence calls. Model-agnostic, genuinely novel process design. Preserve the three-section schema, classification-before-verdict order, and approval gate verbatim. Its core (`process-review` + the ledger schema) is runtime-agnostic; only `stateful-review`'s Step 1 plumbing is environment-coupled. (`generic/ai-agents/reviews/README.md`; `generic/skills/process-review/skill.md:56-68,104-106`)
2. **Exactly-one-owner routing. [HIGH · strength]** Every skill resolves to exactly two states (pinned or defaultModel); `both` was deliberately removed with the rationale documented in code. Keep the invariant even when adding models — extend the enum, never reintroduce a "runs everywhere" state. (`bin/lib.mjs:99-106,398-402`)
3. **Single-source compilation with three per-model variation mechanisms. [strength]** Verified end-to-end: Claude's compiled `plan-task` uses EnterPlanMode; the Codex vars degrade to prose; unknown placeholders warn visibly rather than blanking. (`generic/skills/plan-task/meta.yml`; `bin/lib.mjs:78-84`)
4. **The `fkit:generated` marker as the prune gate. [strength]** Pruning deletes only marker-bearing dirs, only on full compiles; scaffolded roles omit the marker and survive. Caveat discovered in verification: the marker gates *pruning* only — compile overwrites by kit-skill *name* regardless, and `bootstrap --force` bypasses the marker entirely. Extend the marker discipline to bootstrap rather than weakening the compile gate. (`bin/compile-skills.mjs:254-276`)
5. **config.json lifecycle done right. [strength]** Idempotent, append-only rename migrations with no version gating; migrate-before-validate ordering so old projects upgrade for free; one-time manifest seed then config.json authoritative; `config-schema.json` regenerated on every load so an LLM never guesses field semantics; one-line actionable errors that list the valid catalog. (`bin/lib.mjs:114-140,293-393`)
6. **Zero-dependency implementation + hermetic end-to-end tests. [strength]** No deps, `node:test`, real CLI entry points against tmp dirs, sha256 byte-idempotency assertions, tradeoffs documented inside the tests. 66/66 in ~0.6 s, re-run during this review. Do not adopt a framework or YAML library. Add coverage for `parseYaml` and stub content in the same style.
7. **Single-CLI softening + graceful degradation where it exists. [strength]** Stubs carry an explicit missing-CLI fallback; the reviewer's sidekick degradation ("never present a one-reviewer run as full") is exemplary and should be the template for the missing wiki-unavailable clause.
8. **The reviewer↔coder shared-ledger contract. [strength]** Byte-compatible schema on both sides, explicit section ownership, stable finding ids, round numbering, mandatory ledger dedup. The one precisely-specified multi-agent interop in the kit — generalize the pattern (producer→coder task briefs have no schema at all).
9. **The adversarial-pass promotion to a sub-agent** (removes the plugin-cache plumbing) and **the architect's boundary design** (reciprocated by neighbors, operational stub discipline) — both worth back-porting into the generic kit.
10. **The wiki schema/index/log discipline** — compact, learnable, faithful to the Karpathy pattern — *as a schema*. Preserve the design, not the claim that it's proven: zero populated examples exist and the `sources/` directory is specified but never created or used.
11. **npx-on-demand execution** (stale installs still run current kit code — deliberately small blast radius) and **the intent-documenting comment culture** in bin/ and tests/ (why `both` was removed, why renames need no version gating). For a kit maintained largely by LLMs, the why-comments are load-bearing infrastructure.

**Rejected as strengths:** the *two-pass bootstrap flow* (the pause-for-edit idea is sound; the implementation self-blocks without `--force` and `--force` clobbers user content — a defect wearing a design's clothes) and the *task/sprint conventions* (folder names exist; no format or template exists to preserve).

### Task 6 — Improvements (prioritized)

Each with rationale and its main tradeoff. #1–#4 are the ones that most change the general-purpose verdict.

1. **Ship the operating layer: task-brief + sprint-plan templates, one worked example project, an end-to-end operating guide.** Install is minutes; operation must be reverse-engineered — the single biggest adoption blocker. Reuse the two formats that already exist (reviews/README.md ledger template, wiki schema.md) in the example rather than re-authoring. *Tradeoff:* templates ossify unexercised formats, and a dogfooded example must be maintained in lockstep with every skill change or becomes actively misleading.
2. **Make setup non-destructive and the manifest validated.** Marker-gate `--force` overwrites (skip/back up files differing from generated output — the `fkit:generated` convention already exists and is used correctly on the compile side); post-parse manifest validation (required fields non-empty; warn on colonless lines and `#` in values); make manifest-bearing bootstrap onto its own pass-1 output succeed; honor `wiki_path` for vault placement. *Tradeoff:* bootstrap loses its brutally simple regenerate-everything semantics; validation on a hand-rolled parser is more parser code under the no-deps policy.
3. **Pin consumption to releases and fix version coherence.** Bake `#v<VERSION>` (or a commit SHA) into the entry skill's npx commands at release time, or add a project-level `kitRef` in config.json the skill interpolates; make `release.mjs` rewrite `plugin.json` and the README status line and fail on any disagreeing version string; teach the entry skill to prefer a local clone/vendored kit and answer version checks offline. *Tradeoff:* pinning adds an explicit upgrade step and version-skew branching; the current single-channel design guarantees everyone runs the same (albeit unreleased) code.
4. **Make single-model setups first-class; make the model list data-driven.** Immediate: interview detects installed CLIs and suppresses cross-model pin suggestions (the seeded Codex pins are what break Claude-only today). Structural: replace `MODEL_ENUM` and hardcoded compile targets with manifest-declared models carrying invoke prefix, exec command, and output adapter — preserving the exactly-one-owner invariant. *Tradeoff:* a provider registry multiplies the compile/test matrix and dilutes the two-model adversarial-review story that is fkit's most distinctive value.
5. **Derive stub transport from per-skill safety metadata, with least-privilege defaults and quote-safe argument passing.** meta.yml gains `read-only` / `interactive` / `non-delegable` / `native-on-both`; read-only skills compile to plan/read-only exec modes; interactive approval-gated skills become non-delegable; arguments pass via stdin or quoted heredoc; escape the `$` invoke prefix. *Tradeoff:* new authoring burden on every skill; least-privilege breaks currently-working write delegations; non-delegable markings create dead ends single-CLI users hit more often.
6. **Fix the role-extension point.** Support project-local `ai-agents/roles/*.preset.md`; warn loudly at bootstrap on declared roles with no preset; ship coder/reviewer (and architect) presets matching the sample manifest. *Tradeoff:* project-local presets become a second source of truth sync can't migrate; more presets enlarge the single-maintainer coherence surface.
7. **Single-source the omnigent port; harden the ledger contract.** Add omnigent as a third compile target (or at minimum stamp copies with source pointers + a drift-check test); define task-id = basename of the task brief file minus extension, stated identically in both stateful-review skills and reviews/README.md, with a documented fallback for brief-less work; single-source the shared-document schema (or move omnigent's to `<task-id>.shared.md`). *Tradeoff:* couples kit releases to an alpha runtime's vocabulary; a rigid task-id rule fails exactly where reviews are most improvised without its escape hatch.
8. **Verify the omnigent runtime mechanics in-repo, or downgrade the positioning.** Run a recorded smoke test, commit it with a verified-against version note; reconcile the gate_pushes wording across all six configs; apply per-agent sandboxes where supported (read-only for both reviewers; wiki-vault denied to non-wiki agents); state in README whether omnigent is supported or experimental. *Tradeoff:* couples releases to a moving alpha runtime; honest labeling publicly shrinks fkit's claim to the multi-agent category.
9. **State the real scope in README** — one screen: git repo, Node ≥ 18, Claude and/or Codex, network for the npx path, pre-merge coverage, one owner, one project per repo, human-hub topology. The cheapest high-impact fix in this list; it *is* the brief's "state the caveats." *Tradeoff:* visibly narrows the marketed claim; documented constraints tend to calcify.

| Agent | Change | Main tradeoff |
|---|---|---|
| fkit-producer | Add plan-sprint / write-task-brief skills backed by shipped templates; make init conditional with an explicit cold-start branch | Prescribed formats constrain teams with existing PM tooling |
| fkit-coder | Adopt the deterministic task-id rule (brief-file basename; documented fallback) and require stating the resolved ledger path before writing | Rigid rule needs an escape hatch for brief-less hotfixes |

### Task 7 — Positioning

**What fkit actually is:** (1) a zero-dependency compiler that single-sources skills to exactly two CLI targets with per-skill owner routing and shell-out delegation stubs; (2) an `ai-agents/` working structure; (3) a review-ledger discipline. It is a **conventions-and-skill-distribution kit, not a multi-agent framework** — its only shipping cross-agent mechanism is a markdown stub telling an LLM to run a shell command.

- **vs. raw Claude Code** *(hypothesis — competitor claims not verifiable from this repo)*: for a Claude-only team, native CLAUDE.md + skills + subagents + plugin distribution already cover playbooks and updates — fkit itself ships through that native channel. The compile/routing machinery pays only when Codex is also in play. A "conventions-only" install mode (structure + ledger + wiki, no compiler) would serve single-CLI users better.
- **vs. Cursor rules / plain AGENTS.md:** those are static context; fkit adds invocable playbooks, lifecycle structure, and versioned sync — but emits nothing Cursor/Copilot/Gemini can consume.
- **vs. CrewAI / AutoGen / LangGraph** *(hypothesis)*: different category. Those are programmatic runtimes with state, parallelism, budgets, provider abstraction, observability; fkit has no runtime and should position as complementary for human-in-the-loop CLI workflows, not competing.
- **vs. the Karpathy LLM-wiki pattern:** the skeleton (schema/rulebook, four page types, bidirectional links, index catalog, append-only log, self-healing lint) is a coherent, faithful instantiation. But it ships unexercised: `sources/` is specified and never created; the "synthesized knowledge not derivable from code" promise is only partially fed by the pipeline; no populated example exists anywhere — including fkit's own repo, whose `ai-agents/` is a fictional "Nimbus" fixture with no wiki-vault.

**Distinctive value (real):** dual-CLI single-sourcing — author once, pin an owner per skill, get automatic cross-CLI delegation and versioned kit updates that never touch project-authored files; nothing in the comparison set does this. And the review-ledger loop-resistance discipline, which is genuinely novel process IP.

**Do not use fkit when:** you run a single model/CLI (most of the machinery is dead weight); your tooling is Cursor/Copilot/Windsurf/Gemini; you need programmatic orchestration, parallelism, budgets, or observability; you are air-gapped or GitHub-blocked; you don't use git; your team is larger than one approver; or headless auto-approved cross-CLI execution violates your security posture.
