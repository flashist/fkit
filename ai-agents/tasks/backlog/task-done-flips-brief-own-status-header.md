## Context

**`/fkit-task-done` updates the board but not the brief.** Its step 5 flips the sprint-plan Status
cell (and a `**Status:**` line inside the sprint plan's body) to `Ôťů Done`, and re-points links ÔÇö but
it **never touches the moved brief file's own `## Status` field.** So a brief moved into
`ai-agents/tasks/done/` still reads `­čö▓ Backlog` internally, while the board of record reads `Ôťů Done`.

**This is standing board-vs-brief drift produced on every completion.** A `/fkit-status` run finds it
and ÔÇö correctly, per its drift rules ÔÇö renders the row `Ôťů Done` but stamps `waiting on owner` on it,
because two sources disagree about a task's state. The owner then gets handed a phantom decision on a
task that is genuinely finished.

**Currently visible on Sprint 2 tasks 23, 30, 31, 32, 33** (all in `done/`, all with briefs reading
`­čö▓ Backlog`), plus two non-Sprint-2 leftovers (`build-fkit-reconnect-tooling.md`,
`fix-claude-agents-md-placeholder-text.md`). The tasks whose briefs *do* read `Ôťů Done` were closed by
an earlier skill version or hand-edited ÔÇö the inconsistency is the tell.

**This is a sibling to task 22** (`harden-task-movers-against-closed-sprint-link-rot`): both are cases
where a mover updates one record of a task's state and silently leaves another stale. Same class, same
fix philosophy ÔÇö make the mover update *every* place the state is written.

**Scope note ÔÇö no scaffold copy.** The mover skills live only in `claude/skills/`; they are **not**
under `claude/scaffold/` (which ships `AGENTS.md`, `CLAUDE.md`, `ai-agents/`, `universal-rules.md`
only). So the single canonical source to edit is `claude/skills/fkit-task-done/SKILL.md`. The
`.claude/skills/` copy is gitignored and regenerated by init ÔÇö do not edit it.

## What to build

Extend `/fkit-task-done`'s procedure (its step 5, alongside the existing sprint-plan status updates) so
that **after moving the brief into `done/`, it also sets the moved brief's own `## Status` field** to
the terminal marker ÔÇö the exact same `Ôťů Done` value the vocabulary defines, matching the wording the
skill already writes into the board.

Requirements the instruction must impose:

- **Set the brief's `## Status` to `Ôťů Done`** ÔÇö the section directly under the `## Status` heading in
  the moved brief file. Same canonical marker, no variant wording.
- **Idempotent.** Re-running the skill on a brief already reading `Ôťů Done` leaves it byte-identical ÔÇö
  no second marker, no appended line. (A task can legitimately be re-run; the header must converge, not
  accumulate.)
- **A brief with no `## Status` section is left untouched and flagged**, not "fixed" by inventing one.
  The skill reports "no `## Status` field found in `<brief>` ÔÇö board updated, brief header unchanged"
  so the owner knows. Do not fabricate a section the author never wrote.
- **Only the brief just moved** is touched ÔÇö never a sweep across `done/` repairing historical drift.
  (The backfill of the already-drifted 23/30/31/32/33 is a separate manual concern, out of scope here;
  this task stops the *recurrence*.)
- **The skill still makes no commit** ÔÇö this changes what the skill writes, not its commit behavior.

Update the skill's step-7 **report** contract too, so a run states that the brief's own header was set
(or flagged as absent) ÔÇö the owner should see it happened.

## Verification steps

- **Run `/fkit-task-done` on a backlog brief and confirm the moved file's `## Status` reads `Ôťů Done`**
  ÔÇö not `­čö▓ Backlog` ÔÇö after the run.
- **Idempotency:** run it a second time against the same (now-in-`done/`) brief; the `## Status`
  section is byte-identical to after the first run ÔÇö exactly one `Ôťů Done`, no duplication.
- **Missing-section case:** run it on a brief that has no `## Status` heading; confirm the file's body
  is unchanged and the run's report explicitly flags the absent section.
- **Board still correct:** the sprint-plan Status cell and any body `**Status:**` line still flip to
  `Ôťů Done` as before ÔÇö this task adds a write, it must not drop the existing ones.
- **A subsequent `/fkit-status` shows no board-vs-brief drift** for a task closed by the updated skill
  (the row reads `Ôťů Done` with `closed`, not `waiting on owner`).
