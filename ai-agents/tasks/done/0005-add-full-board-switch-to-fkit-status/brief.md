## Context

**`/fkit-status` has no way to force the full board on a repeat call in the same session.** Step 5 of
the skill (`claude/skills/fkit-status/SKILL.md`) says: on a repeat status in the same session, report
the **delta** — what changed — not the whole state. That is a good default; it keeps a second status
call terse. But it means that once the owner has run `/fkit-status` once, there is **no slash-command
path back to the complete dashboard** — subsequent calls delta, and a named sprint
(`/fkit-status Sprint 2`) still counts as a repeat and deltas too.

The full board already exists in the skill — it is **step 4**, the dashboard. Nothing new needs to be
designed; the board, its six-marker vocabulary, its drift reconciliation, and its roll-up all live in
`fkit-status` already. **The only gap is an explicit override** that says "render the complete board
regardless of the delta default."

**Why a switch and not a new skill.** A separate `fkit-sprint-dashboard` skill would either duplicate
step 4's dashboard spec — the most intricate spec the producer owns, and now two copies that drift — or
be a thin wrapper over the same files. It would also cost a new **role-locked** skill registration:
`skills_for_role()` in `claude/fkit-claude.sh` plus the two hand-maintained mirrors (`fkit-team` SKILL,
`claude/README.md`) — the three-way drift surface task 23 exists to guard. A switch on the existing
skill avoids all of that. (Producer/owner discussion, 2026-07-16.)

**No scaffold copy.** The skills live only in `claude/skills/`, not under `claude/scaffold/`, so the
single canonical source to edit is `claude/skills/fkit-status/SKILL.md`. The `.claude/` copy is
gitignored and regenerated by init — do not edit it.

## What to build

Add an explicit **full-board keyword** to `/fkit-status` that forces the complete dashboard even on a
repeat call in the same session.

- **A reserved keyword** in the argument (recommend `full`; the coder may also accept `all` / `board` as
  aliases). When present, the skill renders the **complete** seven-beat status with the **full step-4
  dashboard — every task row**, not a delta.
- **The keyword is reserved and must not be parsed as a sprint name.** `/fkit-status full` targets the
  **active** sprint and forces the full board; `/fkit-status Sprint 2 full` targets Sprint 2 and forces
  the full board. `full` alone is never treated as a sprint name.
- **The delta-on-repeat behavior (step 5) stays the default** — the keyword is an escape hatch, not a
  replacement. Without the keyword, a repeat call still deltas exactly as today.
- **Update the skill's own documentation to match:** the `**Argument:**` contract section, the step-5
  wording (add a clause: *an explicit full-board request overrides the delta default and renders the
  complete board*), and the `## Usage` block (add the new invocations).

This is **skill-instruction text only** — there is no separate parser to write; the skill is prose the
producer agent follows. The change is to that prose.

## Verification steps

- **`/fkit-status full` on a repeat call in the same session renders the complete dashboard** — every
  task row of the active sprint — not a delta. (Reproduce the exact situation that motivated this: run
  `/fkit-status` once, then `/fkit-status full`, and confirm the second call shows all rows.)
- **`/fkit-status Sprint 2 full` renders the full board for Sprint 2**, not a delta and not the active
  sprint if it differs.
- **`full` is never resolved as a sprint name** — `/fkit-status full` does not error with "no sprint
  named `full`" and does not misfile; it targets the active sprint.
- **The plain repeat still deltas** — `/fkit-status` (no keyword) on a repeat call still reports only
  what changed, unchanged from today.
- **The first call of a fresh session is unaffected** — it already renders the full board; the keyword
  does not change that path.
- **The skill's Argument, step-5, and Usage sections all document the keyword** consistently — no
  section left describing the old contract.
